# NON_NEGOTIABLES.md

# Trading Academy Constitution

Version 1.0

---

# Purpose

This document contains the permanent principles of Trading Academy.

These rules override convenience, speed, or personal preference.

If a future implementation conflicts with these principles, these principles take priority.

---

# Rule 1 — Education Comes Before Entertainment

The purpose of every feature is to improve the user's understanding of markets.

Gamification should increase engagement without distracting from learning.

Every game should teach.

Every animation should have a purpose.

Every interaction should reinforce knowledge.

---

# Rule 2 — Risk Management Comes Before Strategy

Users must learn:

• Position sizing

• Capital preservation

• Stop losses

• Risk-to-reward

• Drawdowns

• Portfolio risk

before they focus on maximizing returns.

The platform should never imply that strategy alone creates successful traders.

---

# Rule 3 — Process Over Profit

Winning money does not necessarily mean making a good decision.

Losing money does not necessarily mean making a bad decision.

The application should reward:

• Following a trading plan

• Respecting risk limits

• Journaling consistently

• Executing with discipline

• Reviewing mistakes

rather than simply earning virtual profits.

---

# Rule 4 — Probability Over Prediction

Never present trading as certainty.

Teach users to think in probabilities.

Examples:

Instead of

"This pattern always works."

Teach

"This setup has historically shown favorable probabilities under certain conditions."

---

# Rule 5 — Learn by Doing

Passive reading should be minimized.

Every educational topic should include one or more of:

• Interactive exercises

• Simulations

• Quizzes

• Guided practice

• Market replay

• Visual feedback

• Reflection

The user should actively participate in learning.

---

# Rule 6 — Beginner First

Assume zero prior knowledge.

Introduce concepts gradually.

Explain terminology clearly.

Avoid unnecessary complexity.

Advanced material should build on earlier lessons.

---

# Rule 7 — Never Encourage Gambling

The application must never:

• Encourage reckless risk-taking

• Celebrate oversized gains

• Promote unrealistic expectations

• Glorify leverage

• Suggest guaranteed outcomes

Instead, reinforce:

• Discipline

• Consistency

• Patience

• Long-term thinking

• Continuous learning

---

# Rule 8 — Honest Feedback

Provide constructive feedback.

Explain why a decision was effective or ineffective.

Do not shame users.

Do not exaggerate success.

Do not fabricate certainty.

---

# Rule 9 — Realistic Simulations

Simulators should model real trading conditions whenever practical.

Include concepts such as:

• Slippage

• Commissions

• Bid/ask spread

• Partial fills (where appropriate)

• Market volatility

• Trading sessions

The objective is realism, not false confidence.

---

# Rule 10 — Build Habits

Encourage routines such as:

Daily lessons

Replay practice

Trade journaling

Review sessions

Weekly reflections

Risk checklists

Consistency matters more than intensity.

---

# Rule 11 — Every Feature Must Answer One Question

Before adding a feature, ask:

"How does this help the user become a better trader?"

If the answer is unclear, redesign or remove the feature.

---

# Rule 12 — Simplicity Wins

Avoid unnecessary complexity.

Choose clarity over cleverness.

A beginner should understand the interface without needing instructions.

---

# Rule 13 — Premium User Experience

Every screen should feel:

• Modern

• Responsive

• Fast

• Accessible

• Consistent

• Professional

Users should feel they are using high-quality educational software.

---

# Rule 14 — Accessibility Is Required

The application must support:

Keyboard navigation

Readable typography

Color contrast

Responsive layouts

Screen readers where appropriate

Accessibility is a core requirement, not an optional enhancement.

---

# Rule 15 — Performance Matters

Users should never wait unnecessarily.

Prefer:

• Fast page loads

• Efficient rendering

• Optimized assets

• Lazy loading

• Smooth interactions

Performance is part of the learning experience.

---

# Rule 16 — Modular by Default

Every new feature should be:

Independent

Reusable

Well documented

Easy to extend

Avoid tightly coupling unrelated features.

---

# Rule 17 — No Technical Debt Without Documentation

If a temporary shortcut is required:

• Explain why.

• Document it.

• Add it to a technical debt list.

• Include a proposed solution.

Technical debt should be intentional, visible, and temporary.

---

# Rule 18 — AI Should Be a Mentor

The AI should:

Ask thoughtful questions.

Encourage reflection.

Identify recurring patterns.

Provide personalized guidance.

Support independent thinking.

Avoid acting as an infallible authority.

---

# Rule 19 — Continuous Improvement

Regularly review the project for:

Better architecture

Better UX

Better accessibility

Better educational value

Better performance

Better maintainability

Improvement is ongoing.

---

# Rule 20 — Quality Is the Default

Nothing should be considered complete until it is:

✓ Functional

✓ Polished

✓ Tested

✓ Responsive

✓ Accessible

✓ Documented

✓ Maintainable

✓ Ready for production

---

# Final Principle

Every decision should reinforce this mission:

**Help people become disciplined, knowledgeable, and responsible traders through deliberate practice, thoughtful feedback, and high-quality educational experiences.**

If a feature does not support that mission, it should be reconsidered.
